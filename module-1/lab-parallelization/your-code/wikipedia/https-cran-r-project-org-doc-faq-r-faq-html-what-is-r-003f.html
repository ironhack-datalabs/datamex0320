2.1 What is R?

\n\n

R is a system for statistical computation and graphics. It consists of\na language plus a run-time environment with graphics, a debugger, access\nto certain system functions, and the ability to run programs stored in\nscript files.\n

\n

The design of R has been heavily influenced by two existing languages:\nBecker, Chambers & Wilks’ S (see What is S?) and Sussman’s\nScheme.\nWhereas the resulting language is very similar in appearance to S, the\nunderlying implementation and semantics are derived from Scheme.\nSee What are the differences between R and S?, for further details.\n

\n

The core of R is an interpreted computer language which allows branching\nand looping as well as modular programming using functions. Most of the\nuser-visible functions in R are written in R. It is possible for the\nuser to interface to procedures written in the C, C++, or FORTRAN\nlanguages for efficiency. The R distribution contains functionality for\na large number of statistical procedures. Among these are: linear and\ngeneralized linear models, nonlinear regression models, time series\nanalysis, classical parametric and nonparametric tests, clustering and\nsmoothing. There is also a large set of functions which provide a\nflexible graphical environment for creating various kinds of data\npresentations. Additional modules (“add-on packages”) are available\nfor a variety of specific purposes (see R Add-On Packages).\n

\n

R was initially written by Ross Ihaka\nand Robert Gentleman at the\nDepartment of Statistics of the University of Auckland in Auckland, New\nZealand. In addition, a large group of individuals has contributed to R\nby sending code and bug reports.\n

\n

Since mid-1997 there has been a core group (the “R Core Team”) who can\nmodify the R source code archive. The group currently consists of Doug\nBates, John Chambers, Peter Dalgaard, Robert Gentleman,\nKurt Hornik, Ross Ihaka, Tomas Kalibera, Michael Lawrence, Friedrich Leisch, Uwe Ligges,\nThomas Lumley, Martin Maechler, Martin Morgan, Paul Murrell, Martyn\nPlummer, Brian Ripley, Deepayan Sarkar, Duncan Temple Lang, Luke\nTierney, and Simon Urbanek.\n

\n

R has a home page at https://www.R-project.org/. It is\nfree software\ndistributed under a GNU-style\ncopyleft, and an\nofficial part of the GNU project\n(“GNU S”).\n

2.2 What machines does R run on?

\n\n

R is being developed for the Unix-like, Windows and Mac families of\noperating systems. Support for Mac OS Classic ended with R 1.7.1.\n

\n

The current version of R will configure and build under a number of\ncommon Unix-like (e.g., https://en.wikipedia.org/wiki/Unix-like)\nplatforms including cpu-linux-gnu for the i386, amd64/x86_64,\nalpha, arm, arm64, hppa, mips/mipsel, powerpc, s390x and sparc\nCPUs (e.g.,\nhttps://buildd.debian.org/build.php?&pkg=r-base), i386-hurd-gnu,\ncpu-kfreebsd-gnu for i386 and amd64,\ni386-pc-solaris, rs6000-ibm-aix, sparc-sun-solaris, x86_64-apple-darwin,\nx86_64-unknown-freebsd and x86_64-unknown-openbsd.\n

\n\n

If you know about other platforms, please drop us a note.\n

2.3 What is the current version of R?

\n \n

R uses a ‘major.minor.patchlevel’ numbering scheme. Based on this,\nthere are the current release version of R (‘r-release’) as well as two\ndevelopment versions of R, a patched version of the current release\n(‘r-patched’) and one working towards the next minor or eventually major\n(‘r-devel’) releases of R, respectively. New features are typically\nintroduced in r-devel, while r-patched is for bug fixes mostly.\n

\n

See https://CRAN.R-project.org/sources.html for the current\nversions of r-release, r-patched and r-devel.\n

2.5.1 How can R be installed (Unix-like)

\n\n

If R is already installed, it can be started by typing R at the\nshell prompt (of course, provided that the executable is in your path).\n

\n

If binaries are available for your platform (see Are there Unix-like binaries for R?), you can use these, following the\ninstructions that come with them.\n

\n

Otherwise, you can compile and install R yourself, which can be done\nvery easily under a number of common Unix-like platforms (see What machines does R run on?). The file INSTALL that comes with the\nR distribution contains a brief introduction, and the “R Installation\nand Administration” guide (see What documentation exists for R?)\nhas full details.\n

\n

Note that you need a FORTRAN compiler or perhaps f2c in\naddition to a C compiler to build R.\n

\n

In the simplest case, untar the R source code, change to the directory\nthus created, and issue the following commands (at the shell prompt):\n

\n
\n
$ ./configure\n$ make\n
\n\n

If these commands execute successfully, the R binary and a shell script\nfront-end called R are created and copied to the bin\ndirectory. You can copy the script to a place where users can invoke\nit, for example to /usr/local/bin. In addition, plain text help\npages as well as HTML and LaTeX versions of the documentation are\nbuilt.\n

\n

Use make dvi to create DVI versions of the R manuals, such as\nrefman.dvi (an R object reference index) and R-exts.dvi,\nthe “R Extension Writers Guide”, in the doc/manual\nsubdirectory. These files can be previewed and printed using standard\nprograms such as xdvi and dvips. You can also use\nmake pdf to build PDF (Portable Document Format) version of the\nmanuals, and view these using e.g. Acrobat. Manuals written in the\nGNU Texinfo system can also be converted to info files\nsuitable for reading online with Emacs or stand-alone GNU\nInfo; use make info to create these versions (note that this\nrequires Makeinfo version 4.5).\n

\n

Finally, use make check to find out whether your R system works\ncorrectly.\n

\n

You can also perform a “system-wide” installation using make\ninstall. By default, this will install to the following directories:\n

\n
\n
${prefix}/bin
\n

the front-end shell script\n

\n
${prefix}/man/man1
\n

the man page\n

\n
${prefix}/lib/R
\n

all the rest (libraries, on-line help system, …). This is the “R\nHome Directory” (R_HOME) of the installed system.\n

\n
\n\n

In the above, prefix is determined during configuration\n(typically /usr/local) and can be set by running\nconfigure with the option\n

\n
\n
$ ./configure --prefix=/where/you/want/R/to/go\n
\n\n

(E.g., the R executable will then be installed into\n/where/you/want/R/to/go/bin.)\n

\n

To install DVI, info and PDF versions of the manuals, use make\ninstall-dvi, make install-info and make install-pdf,\nrespectively.\n

2.5.2 How can R be installed (Windows)

\n\n

The bin/windows directory of a CRAN site contains binaries for\na base distribution and add-on packages from CRAN to run on Windows\n7 and later (including 64-bit versions of Windows) on ix86 and x86_64\nchips. The Windows version of R was created by Robert Gentleman and\nGuido Masarotto, Brian D. Ripley and Duncan Murdoch made substantial \ncontributions and it is now being maintained by\nother members of the R Core team.\n

\n

The same directory has links to snapshots of the r-patched and r-devel\nversions of R.\n

\n

See the “R for Windows FAQ” for more details.\n

Debian packages, maintained by Dirk Eddelbuettel, have long been part of\nthe Debian distribution, and can be accessed through APT, the Debian\npackage maintenance tool. Use e.g. apt-get install r-base\nr-recommended to install the R environment and recommended packages.\nIf you also want to build R packages from source, also run apt-get\ninstall r-base-dev to obtain the additional tools required for this.\nSo-called “backports” of the current R packages for at least the\nstable distribution of Debian are provided by Johannes Ranke, and\navailable from CRAN. Native backports for Ubuntu are\nprovided by Michael Rutter.\n

\n

R binaries for Fedora, maintained by Tom “Spot” Callaway, are provided\nas part of the Fedora distribution and can be accessed through\nyum, the RPM installer/updater. Note that the “Software”\napplication (gnome-software), which is the default GUI for\nsoftware installation in Fedora 20, cannot be used to install R. It is\ntherefore recommended to use the yum command line tool.\nThe Fedora R RPM is a “meta-package” which installs all the user and\ndeveloper components of R (available separately as R-core and\nR-devel), as well as R-java, which ensures that R is\nconfigured for use with Java. The R RPM also installs the standalone R\nmath library (libRmath and libRmath-devel), although this\nis not necessary to use R. When a new version of R is released, there\nmay be a delay of up to 2 weeks until the Fedora RPM becomes publicly\navailable, as it must pass through the statutory Fedora review process.\nRPMs for a selection of R packages are also provided by Fedora. The\nExtra Packages for Enterprise Linux (EPEL) project\n(https://fedoraproject.org/wiki/EPEL) provides ports of the Fedora\nRPMs for RedHat Enterprise Linux and compatible distributions (e.g.,\nCentos, Scientific Linux, Oracle Linux).\n

\n

See https://CRAN.R-project.org/bin/linux/suse/README.html for\ninformation about RPMs for openSUSE.\n

\n

No other binary distributions are currently publically available via\nCRAN.\n

2.7 What documentation exists for R?

\n\n

Online documentation for most of the functions and variables in R\nexists, and can be printed on-screen by typing help(name)\n(or ?name) at the R prompt, where name is the name of\nthe topic help is sought for. (In the case of unary and binary\noperators and control-flow special forms, the name may need to be be\nquoted.)\n

\n

This documentation can also be made available as one reference manual\nfor on-line reading in HTML and PDF formats, and as hardcopy via\nLaTeX, see How can R be installed?. An up-to-date HTML\nversion is always available for web browsing at\nhttps://stat.ethz.ch/R-manual/.\n

\n\n

The R distribution also comes with the following manuals.\n

\n
    \n
  • “An Introduction to R” (R-intro)\nincludes information on data types, programming elements, statistical\nmodeling and graphics. This document is based on the “Notes on\nS-PLUS” by Bill Venables and David Smith.\n
  • “Writing R Extensions” (R-exts)\ncurrently describes the process of creating R add-on packages, writing R\ndocumentation, R’s system and foreign language interfaces, and the R\nAPI.\n
  • “R Data Import/Export” (R-data)\nis a guide to importing and exporting data to and from R.\n
  • “The R Language Definition” (R-lang),\na first version of the “Kernighan & Ritchie of R”, explains\nevaluation, parsing, object oriented programming, computing on the\nlanguage, and so forth.\n

3.3 What are the differences between R and S?

\n\n

We can regard S as a language with three current implementations or\n“engines”, the “old S engine” (S version 3; S-PLUS 3.x and 4.x),\nthe “new S engine” (S version 4; S-PLUS 5.x and above), and R.\nGiven this understanding, asking for “the differences between R and S”\nreally amounts to asking for the specifics of the R implementation of\nthe S language, i.e., the difference between the R and S engines.\n

\n

For the remainder of this section, “S” refers to the S engines and not\nthe S language.\n

3.3.1 Lexical scoping

\n\n

Contrary to other implementations of the S language, R has adopted an\nevaluation model in which nested function definitions are lexically\nscoped. This is analogous to the evaluation model in Scheme.\n

\n

This difference becomes manifest when free variables occur in a\nfunction. Free variables are those which are neither formal parameters\n(occurring in the argument list of the function) nor local variables\n(created by assigning to them in the body of the function). In S, the\nvalues of free variables are determined by a set of global variables\n(similar to C, there is only local and global scope). In R, they are\ndetermined by the environment in which the function was created.\n

\n

Consider the following function:\n

\n
\n
cube <- function(n) {\n  sq <- function() n * n\n  n * sq()\n}\n
\n\n

Under S, sq() does not “know” about the variable n\nunless it is defined globally:\n

\n
\n
S> cube(2)\nError in sq():  Object "n" not found\nDumped\nS> n <- 3\nS> cube(2)\n[1] 18\n
\n\n

In R, the “environment” created when cube() was invoked is\nalso looked in:\n

\n
\n
R> cube(2)\n[1] 8\n
\n\n\n\n\n\n\n\n

As a more “interesting” real-world problem, suppose you want to write\na function which returns the density function of the r-th order\nstatistic from a sample of size n from a (continuous)\ndistribution. For simplicity, we shall use both the cdf and pdf of the\ndistribution as explicit arguments. (Example compiled from various\npostings by Luke Tierney.)\n

\n

The S-PLUS documentation for call() basically suggests the\nfollowing:\n

\n
\n
dorder <- function(n, r, pfun, dfun) {\n  f <- function(x) NULL\n  con <- round(exp(lgamma(n + 1) - lgamma(r) - lgamma(n - r + 1)))\n  PF <- call(substitute(pfun), as.name("x"))\n  DF <- call(substitute(dfun), as.name("x"))\n  f[[length(f)]] <-\n    call("*", con,\n         call("*", call("^", PF, r - 1),\n              call("*", call("^", call("-", 1, PF), n - r),\n                   DF)))\n  f\n}\n
\n\n

Rather tricky, isn’t it? The code uses the fact that in S,\nfunctions are just lists of special mode with the function body as the\nlast argument, and hence does not work in R (one could make the idea\nwork, though).\n

\n

A version which makes heavy use of substitute() and seems to work\nunder both S and R is\n

\n
\n
dorder <- function(n, r, pfun, dfun) {\n  con <- round(exp(lgamma(n + 1) - lgamma(r) - lgamma(n - r + 1)))\n  eval(substitute(function(x) K * PF(x)^a * (1 - PF(x))^b * DF(x),\n                  list(PF = substitute(pfun), DF = substitute(dfun),\n                       a = r - 1, b = n - r, K = con)))\n}\n
\n\n

(the eval() is not needed in S).\n

\n

However, in R there is a much easier solution:\n

\n
\n
dorder <- function(n, r, pfun, dfun) {\n  con <- round(exp(lgamma(n + 1) - lgamma(r) - lgamma(n - r + 1)))\n  function(x) {\n    con * pfun(x)^(r - 1) * (1 - pfun(x))^(n - r) * dfun(x)\n  }\n}\n
\n\n

This seems to be the “natural” implementation, and it works because\nthe free variables in the returned function can be looked up in the\ndefining environment (this is lexical scope).\n

\n

Note that what you really need is the function closure, i.e., the\nbody along with all variable bindings needed for evaluating it. Since\nin the above version, the free variables in the value function are not\nmodified, you can actually use it in S as well if you abstract out the\nclosure operation into a function MC() (for “make closure”):\n

\n
\n
dorder <- function(n, r, pfun, dfun) {\n  con <- round(exp(lgamma(n + 1) - lgamma(r) - lgamma(n - r + 1)))\n  MC(function(x) {\n       con * pfun(x)^(r - 1) * (1 - pfun(x))^(n - r) * dfun(x)\n     },\n     list(con = con, pfun = pfun, dfun = dfun, r = r, n = n))\n}\n
\n\n

Given the appropriate definitions of the closure operator, this works in\nboth R and S, and is much “cleaner” than a substitute/eval solution\n(or one which overrules the default scoping rules by using explicit\naccess to evaluation frames, as is of course possible in both R and S).\n

\n

For R, MC() simply is\n

\n
\n
MC <- function(f, env) f\n
\n\n

(lexical scope!), a version for S is\n

\n
\n
MC <- function(f, env = NULL) {\n  env <- as.list(env)\n  if (mode(f) != "function")\n    stop(paste("not a function:", f))\n  if (length(env) > 0 && any(names(env) == ""))\n    stop(paste("not all arguments are named:", env))\n  fargs <- if(length(f) > 1) f[1:(length(f) - 1)] else NULL\n  fargs <- c(fargs, env)\n  if (any(duplicated(names(fargs))))\n    stop(paste("duplicated arguments:", paste(names(fargs)),\n         collapse = ", "))\n  fbody <- f[length(f)]\n  cf <- c(fargs, fbody)\n  mode(cf) <- "function"\n  return(cf)\n}\n
\n\n

Similarly, most optimization (or zero-finding) routines need some\narguments to be optimized over and have other parameters that depend on\nthe data but are fixed with respect to optimization. With R scoping\nrules, this is a trivial problem; simply make up the function with the\nrequired definitions in the same environment and scoping takes care of\nit. With S, one solution is to add an extra parameter to the function\nand to the optimizer to pass in these extras, which however can only\nwork if the optimizer supports this.\n

\n

Nested lexically scoped functions allow using function closures and\nmaintaining local state. A simple example (taken from Abelson and\nSussman) is obtained by typing demo("scoping") at the R prompt.\nFurther information is provided in the standard R reference “R: A\nLanguage for Data Analysis and Graphics” (see What documentation exists for R?) and in Robert Gentleman and Ross Ihaka (2000), “Lexical\nScope and Statistical Computing”,\nJournal of\nComputational and Graphical Statistics, 9, 491–508.\n

\n

Nested lexically scoped functions also imply a further major difference.\nWhereas S stores all objects as separate files in a directory somewhere\n(usually .Data under the current directory), R does not. All\nobjects in R are stored internally. When R is started up it grabs a\npiece of memory and uses it to store the objects. R performs its own\nmemory management of this piece of memory, growing and shrinking its\nsize as needed. Having everything in memory is necessary because it is\nnot really possible to externally maintain all relevant “environments”\nof symbol/value pairs. This difference also seems to make R\nfaster than S.\n

\n

The down side is that if R crashes you will lose all the work for the\ncurrent session. Saving and restoring the memory “images” (the\nfunctions and data stored in R’s internal memory at any time) can be a\nbit slow, especially if they are big. In S this does not happen,\nbecause everything is saved in disk files and if you crash nothing is\nlikely to happen to them. (In fact, one might conjecture that the S\ndevelopers felt that the price of changing their approach to persistent\nstorage just to accommodate lexical scope was far too expensive.)\nHence, when doing important work, you might consider saving often (see\nHow can I save my workspace?) to safeguard against possible\ncrashes. Other possibilities are logging your sessions, or have your R\ncommands stored in text files which can be read in using\nsource().\n

\n
\n

Note: If you run R from within Emacs (see R and Emacs), you can save the\ncontents of the interaction buffer to a file and conveniently manipulate\nit using ess-transcript-mode, as well as save source copies of\nall functions and data used.\n

\n\n
\n

3.3.2 Models

\n\n

There are some differences in the modeling code, such as\n

\n
    \n
  • Whereas in S, you would use lm(y ~ x^3) to regress y on\nx^3, in R, you have to insulate powers of numeric vectors (using\nI()), i.e., you have to use lm(y ~ I(x^3)).\n
  • The glm family objects are implemented differently in R and S. The same\nfunctionality is available but the components have different names.\n
  • Option na.action is set to "na.omit" by default in R,\nbut not set in S.\n
  • Terms objects are stored differently. In S a terms object is an\nexpression with attributes, in R it is a formula with attributes. The\nattributes have the same names but are mostly stored differently.\n
  • Finally, in R y ~ x + 0 is an alternative to y ~ x - 1 for\nspecifying a model with no intercept. Models with no parameters at all\ncan be specified by y ~ 0.\n
\n\n
\n
\n

3.3.3 Others

\n\n

Apart from lexical scoping and its implications, R follows the S\nlanguage definition in the Blue and White Books as much as possible, and\nhence really is an “implementation” of S. There are some intentional\ndifferences where the behavior of S is considered “not clean”. In\ngeneral, the rationale is that R should help you detect programming\nerrors, while at the same time being as compatible as possible with S.\n

\n

Some known differences are the following.\n

\n
    \n
  • In R, if x is a list, then x[i] <- NULL and x[[i]]\n<- NULL remove the specified elements from x. The first of\nthese is incompatible with S, where it is a no-op. (Note that you can\nset elements to NULL using x[i] <- list(NULL).)\n\n\n
  • In S, the functions named .First and .Last in the\n.Data directory can be used for customizing, as they are executed\nat the very beginning and end of a session, respectively.\n\n

    In R, the startup mechanism is as follows. Unless --no-environ\nwas given on the command line, R searches for site and user files to\nprocess for setting environment variables. Then, R searches for a\nsite-wide startup profile unless the command line option\n--no-site-file was given. This code is loaded in package\nbase. Then, unless --no-init-file was given, R\nsearches for a user profile file, and sources it into the user\nworkspace. It then loads a saved image of the user workspace from\n.RData in case there is one (unless --no-restore-data or\n--no-restore were specified). Next, a function .First()\nis run if found on the search path. Finally, function .First.sys\nin the base package is run. When terminating an R session, by\ndefault a function .Last is run if found on the search path,\nfollowed by .Last.sys. If needed, the functions .First()\nand .Last() should be defined in the appropriate startup\nprofiles. See the help pages for .First and .Last for\nmore details.\n

    \n
  • In R, T and F are just variables being set to TRUE\nand FALSE, respectively, but are not reserved words as in S and\nhence can be overwritten by the user. (This helps e.g. when you have\nfactors with levels "T" or "F".) Hence, when writing code\nyou should always use TRUE and FALSE.\n\n
  • In R, dyn.load() can only load shared objects, as created\nfor example by R CMD SHLIB.\n\n
  • In R, attach() currently only works for lists and data frames,\nbut not for directories. (In fact, attach() also works for R\ndata files created with save(), which is analogous to attaching\ndirectories in S.) Also, you cannot attach at position 1.\n\n
  • Categories do not exist in R, and never will as they are deprecated now\nin S. Use factors instead.\n\n
  • In R, For() loops are not necessary and hence not supported.\n\n
  • In R, assign() uses the argument envir= rather than\nwhere= as in S.\n\n
  • The random number generators are different, and the seeds have different\nlength.\n\n
  • R passes integer objects to C as int * rather than long *\nas in S.\n\n
  • R has no single precision storage mode. However, as of version 0.65.1,\nthere is a single precision interface to C/FORTRAN subroutines.\n\n
  • By default, ls() returns the names of the objects in the current\n(under R) and global (under S) environment, respectively. For example,\ngiven\n\n
    \n
    x <- 1; fun <- function() {y <- 1; ls()}\n
    \n\n

    then fun() returns "y" in R and "x" (together with\nthe rest of the global environment) in S.\n

    \n
  • R allows for zero-extent matrices (and arrays, i.e., some elements of\nthe dim attribute vector can be 0). This has been determined a\nuseful feature as it helps reducing the need for special-case tests for\nempty subsets. For example, if x is a matrix, x[, FALSE]\nis not NULL but a “matrix” with 0 columns. Hence, such objects\nneed to be tested for by checking whether their length() is zero\n(which works in both R and S), and not using is.null().\n\n
  • Named vectors are considered vectors in R but not in S (e.g.,\nis.vector(c(a = 1:3)) returns FALSE in S and TRUE\nin R).\n\n
  • Data frames are not considered as matrices in R (i.e., if DF is a\ndata frame, then is.matrix(DF) returns FALSE in R and\nTRUE in S).\n\n
  • R by default uses treatment contrasts in the unordered case, whereas S\nuses the Helmert ones. This is a deliberate difference reflecting the\nopinion that treatment contrasts are more natural.\n\n
  • In R, the argument of a replacement function which corresponds to the\nright hand side must be named ‘value’. E.g., f(a) <- b is\nevaluated as a <- "f<-"(a, value = b). S always takes the last\nargument, irrespective of its name.\n\n
  • In S, substitute() searches for names for substitution in the\ngiven expression in three places: the actual and the default arguments\nof the matching call, and the local frame (in that order). R looks in\nthe local frame only, with the special rule to use a “promise” if a\nvariable is not evaluated. Since the local frame is initialized with\nthe actual arguments or the default expressions, this is usually\nequivalent to S, until assignment takes place.\n\n
  • In S, the index variable in a for() loop is local to the inside\nof the loop. In R it is local to the environment where the for()\nstatement is executed.\n\n
  • In S, tapply(simplify=TRUE) returns a vector where R returns a\none-dimensional array (which can have named dimnames).\n\n
  • In S(-PLUS) the C locale is used, whereas in R the current\noperating system locale is used for determining which characters are\nalphanumeric and how they are sorted. This affects the set of valid\nnames for R objects (for example accented chars may be allowed in R) and\nordering in sorts and comparisons (such as whether "aA" < "Bb" is\ntrue or false). From version 1.2.0 the locale can be (re-)set in R by\nthe Sys.setlocale() function.\n\n
  • In S, missing(arg) remains TRUE if arg is\nsubsequently modified; in R it doesn’t.\n\n
  • From R version 1.3.0, data.frame strips I() when creating\n(column) names.\n\n
  • In R, the string "NA" is not treated as a missing value in a\ncharacter variable. Use as.character(NA) to create a missing\ncharacter value.\n\n
  • R disallows repeated formal arguments in function calls.\n\n
  • In S, dump(), dput() and deparse() are essentially\ndifferent interfaces to the same code. In R from version 2.0.0, this is\nonly true if the same control argument is used, but by default it\nis not. By default dump() tries to write code that will evaluate\nto reproduce the object, whereas dput() and deparse()\ndefault to options for producing deparsed code that is readable.\n\n
  • In R, indexing a vector, matrix, array or data frame with [ using\na character vector index looks only for exact matches (whereas [[\nand $ allow partial matches). In S, [ allows partial\nmatches.\n\n
  • S has a two-argument version of atan and no atan2. A call \nin S such as atan(x1, x2) is equivalent to R’s atan2(x1, \nx2). However, beware of named arguments since S’s atan(x = a, y\n= b) is equivalent to R’s atan2(y = a, x = b) with the meanings\nof x and y interchanged. (R used to have undocumented\nsupport for a two-argument atan with positional arguments, but\nthis has been withdrawn to avoid further confusion.)\n\n
  • Numeric constants with no fractional and exponent (i.e., only integer)\npart are taken as integer in S-PLUS 6.x or later, but as double in R.\n\n
\n\n

There are also differences which are not intentional, and result from\nmissing or incorrect code in R. The developers would appreciate hearing\nabout any deficiencies you may find (in a written report fully\ndocumenting the difference as you see it). Of course, it would be\nuseful if you were to implement the change yourself and make sure it\nworks.\n

3.4 Is there anything R can do that S-PLUS cannot?

5.2 How can add-on packages be installed?

\n\n

(Unix-like only.) The add-on packages on CRAN come as gzipped tar\nfiles named pkg_version.tar.gz, which may in fact be\n“bundles” containing more than one package. Let path be the\npath to such a package file. Provided that tar and\ngzip are available on your system, type\n

\n
\n
$ R CMD INSTALL path/pkg_version.tar.gz\n
\n\n

at the shell prompt to install to the library tree rooted at the first\ndirectory in your library search path (see the help page for\n.libPaths() for details on how the search path is determined).\n

\n

To install to another tree (e.g., your private one), use\n

\n
\n
$ R CMD INSTALL -l lib path/pkg_version.tar.gz\n
\n\n

where lib gives the path to the library tree to install to.\n

\n

Even more conveniently, you can install and automatically update\npackages from within R if you have access to repositories such as\nCRAN. See the help page for available.packages() for more\ninformation.\n

6.1 Is there Emacs support for R?

\n\n

There is an Emacs package called ESS (“Emacs Speaks\nStatistics”) which provides a standard interface between statistical\nprograms and statistical processes. It is intended to provide\nassistance for interactive statistical programming and data analysis.\nLanguages supported include: S dialects (R, S 3/4, and S-PLUS\n3.x/4.x/5.x/6.x/7.x), LispStat dialects (XLispStat, ViSta), SAS, Stata,\nand BUGS.\n

\n

ESS grew out of the need for bug fixes and extensions to\nS-mode 4.8 (which was a GNU Emacs interface to S/S-PLUS\nversion 3 only). The current set of developers desired support for\nXEmacs, R, S4, and MS Windows. In addition, with new modes being\ndeveloped for R, Stata, and SAS, it was felt that a unifying interface\nand framework for the user interface would benefit both the user and the\ndeveloper, by helping both groups conform to standard Emacs usage. The\nend result is an increase in efficiency for statistical programming and\ndata analysis, over the usual tools.\n

\n

R support contains code for editing R source code (syntactic indentation\nand highlighting of source code, partial evaluations of code, loading\nand error-checking of code, and source code revision maintenance) and\ndocumentation (syntactic indentation and highlighting of source code,\nsending examples to running ESS process, and previewing),\ninteracting with an inferior R process from within Emacs (command-line\nediting, searchable command history, command-line completion of R object\nand file names, quick access to object and search lists, transcript\nrecording, and an interface to the help system), and transcript\nmanipulation (recording and saving transcript files, manipulating and\nediting saved transcripts, and re-evaluating commands from transcript\nfiles).\n

\n

The latest stable version of ESS is available via CRAN or\nthe ESS web page.\n

\n

ESS comes with detailed installation instructions.\n

\n

For help with ESS, send email to\nESS-help@r-project.org.\n

\n

Please send bug reports and suggestions on ESS to\nESS-bugs@r-project.org. The easiest way to do this from is\nwithin Emacs by typing M-x ess-submit-bug-report or using the\n[ESS] or [iESS] pulldown menus.\n

6.2 Should I run R from within Emacs?

\n\n

Yes, instead of just running it in a console, definitely. As an\nalternative to other IDE’s such as Rstudio, possibly, notably if you\nare interested to use Emacs for other computer interaction.\nYou’d be using ESS, Emacs Speaks Statistics, see previous FAQ.\n

\n

Inferior R mode provides a readline/history\nmechanism, object name completion, and syntax-based highlighting of the\ninteraction buffer using Font Lock mode, as well as a very convenient\ninterface to the R help system.\n

\n

Of course, it also integrates nicely with the mechanisms for editing R\nsource using Emacs. One can write code in one Emacs buffer and send\nwhole or parts of it for execution to R; this is helpful for both data\nanalysis and programming. One can also seamlessly integrate with a\nrevision control system, in order to maintain a log of changes in your\nprograms and data, as well as to allow for the retrieval of past\nversions of the code.\n

\n

In addition, it allows you to keep a record of your session, which can\nalso be used for error recovery through the use of the transcript mode.\n

\n

To specify command line arguments for the inferior R process, use\nC-u M-x R for starting R.\n

6.3 Debugging R from within Emacs

\n\n

To debug R “from within Emacs”, there are several possibilities. To\nuse the Emacs GUD (Grand Unified Debugger) library with the recommended\ndebugger GDB, type M-x gdb and give the path to the R\nbinary as argument. At the gdb prompt, set\nR_HOME and other environment variables as needed (using e.g.\nset env R_HOME /path/to/R/, but see also below), and start the\nbinary with the desired arguments (e.g., run --quiet).\n

\n

If you have ESS, you can do C-u M-x R RET - d\nSPC g d b RET to start an inferior R process with arguments\n-d gdb.\n

\n

A third option is to start an inferior R process via ESS\n(M-x R) and then start GUD (M-x gdb) giving the R binary\n(using its full path name) as the program to debug. Use the program\nps to find the process number of the currently running R\nprocess then use the attach command in gdb to attach it to that\nprocess. One advantage of this method is that you have separate\n*R* and *gud-gdb* windows. Within the *R* window\nyou have all the ESS facilities, such as object-name\ncompletion, that we know and love.\n

\n

When using GUD mode for debugging from within Emacs, you may find it\nmost convenient to use the directory with your code in it as the current\nworking directory and then make a symbolic link from that directory to\nthe R binary. That way .gdbinit can stay in the directory with\nthe code and be used to set up the environment and the search paths for\nthe source, e.g. as follows:\n

\n
\n
set env R_HOME /opt/R\nset env R_PAPERSIZE letter\nset env R_PRINTCMD lpr\ndir /opt/R/src/appl\ndir /opt/R/src/main\ndir /opt/R/src/nmath\ndir /opt/R/src/unix\n

7.1 How can I set components of a list to NULL?

\n\n

You can use\n

\n
\n
x[i] <- list(NULL)\n
\n\n

to set component i of the list x to NULL, similarly\nfor named components. Do not set x[i] or x[[i]] to\nNULL, because this will remove the corresponding component from\nthe list.\n

\n

For dropping the row names of a matrix x, it may be easier to use\nrownames(x) <- NULL, similarly for column names.\n

7.4 How can I get eval() and D() to work?

\n\n

Strange things will happen if you use eval(print(x), envir = e)\nor D(x^2, "x"). The first one will either tell you that\n"x" is not found, or print the value of the wrong x.\nThe other one will likely return zero if x exists, and an error\notherwise.\n

\n

This is because in both cases, the first argument is evaluated in the\ncalling environment first. The result (which should be an object of\nmode "expression" or "call") is then evaluated or\ndifferentiated. What you (most likely) really want is obtained by\n“quoting” the first argument upon surrounding it with\nexpression(). For example,\n

\n
\n
R> D(expression(x^2), "x")\n2 * x\n
\n\n

Although this behavior may initially seem to be rather strange, it is\nperfectly logical. The “intuitive” behavior could easily be\nimplemented, but problems would arise whenever the expression is\ncontained in a variable, passed as a parameter, or is the result of a\nfunction call. Consider for instance the semantics in cases like\n

\n
\n
D2 <- function(e, n) D(D(e, n), n)\n
\n\n

or\n

\n
\n
g <- function(y) eval(substitute(y), sys.frame(sys.parent(n = 2)))\ng(a * b)\n
\n\n

See the help page for deriv() for more examples.\n

\n

7.5 Why do my matrices lose dimensions?

\n\n

When a matrix with a single row or column is created by a subscripting\noperation, e.g., row <- mat[2, ], it is by default turned into a\nvector. In a similar way if an array with dimension, say, 2 x 3 x 1 x 4 is created by subscripting it will be coerced into a 2 x 3 x 4\narray, losing the unnecessary dimension. After much discussion this has\nbeen determined to be a feature.\n

\n

To prevent this happening, add the option drop = FALSE to the\nsubscripting. For example,\n

\n
\n
rowmatrix <- mat[2, , drop = FALSE]  # creates a row matrix\ncolmatrix <- mat[, 2, drop = FALSE]  # creates a column matrix\na <- b[1, 1, 1, drop = FALSE]        # creates a 1 x 1 x 1 array\n
\n\n

The drop = FALSE option should be used defensively when\nprogramming. For example, the statement\n

\n
\n
somerows <- mat[index, ]\n
\n\n

will return a vector rather than a matrix if index happens to\nhave length 1, causing errors later in the code. It should probably be\nrewritten as\n

\n
\n
somerows <- mat[index, , drop = FALSE]\n
\n\n
\n

7.6 How does autoloading work?

\n\n

R has a special environment called .AutoloadEnv. Using\nautoload(name, pkg), where name and\npkg are strings giving the names of an object and the package\ncontaining it, stores some information in this environment. When R\ntries to evaluate name, it loads the corresponding package\npkg and reevaluates name in the new package’s\nenvironment.\n

\n

Using this mechanism makes R behave as if the package was loaded, but\ndoes not occupy memory (yet).\n

\n

See the help page for autoload() for a very nice example.\n

7.7 How should I set options?

\n\n

The function options() allows setting and examining a variety of\nglobal “options” which affect the way in which R computes and displays\nits results. The variable .Options holds the current values of\nthese options, but should never directly be assigned to unless you want\nto drive yourself crazy—simply pretend that it is a “read-only”\nvariable.\n

\n

For example, given\n

\n
\n
test1 <- function(x = pi, dig = 3) {\n  oo <- options(digits = dig); on.exit(options(oo));\n  cat(.Options$digits, x, "\\n")\n}\ntest2 <- function(x = pi, dig = 3) {\n  .Options$digits <- dig\n  cat(.Options$digits, x, "\\n")\n}\n
\n\n

we obtain:\n

\n
\n
R> test1()\n3 3.14 \nR> test2()\n3 3.141593\n
\n\n

What is really used is the global value of .Options, and\nusing options(OPT = VAL) correctly updates it. Local copies of\n.Options, either in .GlobalEnv or in a function\nenvironment (frame), are just silently disregarded.\n

\n
\n

7.8 How do file names work in Windows?

\n\n

As R uses C-style string handling, ‘\\’ is treated as an escape\ncharacter, so that for example one can enter a newline as ‘\\n’.\nWhen you really need a ‘\\’, you have to escape it with another\n‘\\’.\n

\n

Thus, in filenames use something like "c:\\\\data\\\\money.dat". You\ncan also replace ‘\\’ by ‘/’ ("c:/data/money.dat").\n

\n

7.9 Why does plotting give a color allocation error?

\n\n

On an X11 device, plotting sometimes, e.g., when running\ndemo("image"), results in “Error: color allocation error”.\nThis is an X problem, and only indirectly related to R. It occurs when\napplications started prior to R have used all the available colors.\n(How many colors are available depends on the X configuration; sometimes\nonly 256 colors can be used.)\n

\n

One application which is notorious for “eating” colors is Netscape.\nIf the problem occurs when Netscape is running, try (re)starting it with\neither the -no-install (to use the default colormap) or the\n-install (to install a private colormap) option.\n

\n

You could also set the colortype of X11() to\n"pseudo.cube" rather than the default "pseudo". See the\nhelp page for X11() for more information.\n

7.10 How do I convert factors to numeric?

\n\n

It may happen that when reading numeric data into R (usually, when\nreading in a file), they come in as factors. If f is such a\nfactor object, you can use\n

\n
\n
as.numeric(as.character(f))\n
\n\n

to get the numbers back. More efficient, but harder to remember, is\n

\n
\n
as.numeric(levels(f))[as.integer(f)]\n
\n\n

In any case, do not call as.numeric() or their likes directly for\nthe task at hand (as as.numeric() or unclass() give the\ninternal codes).\n

7.12 What are the enclosing and parent environments?

\n\n

Inside a function you may want to access variables in two additional\nenvironments: the one that the function was defined in (“enclosing”),\nand the one it was invoked in (“parent”).\n

\n

If you create a function at the command line or load it in a package its\nenclosing environment is the global workspace. If you define a function\nf() inside another function g() its enclosing environment\nis the environment inside g(). The enclosing environment for a\nfunction is fixed when the function is created. You can find out the\nenclosing environment for a function f() using\nenvironment(f).\n

\n

The “parent” environment, on the other hand, is defined when you\ninvoke a function. If you invoke lm() at the command line its\nparent environment is the global workspace, if you invoke it inside a\nfunction f() then its parent environment is the environment\ninside f(). You can find out the parent environment for an\ninvocation of a function by using parent.frame() or\nsys.frame(sys.parent()).\n

\n

So for most user-visible functions the enclosing environment will be the\nglobal workspace, since that is where most functions are defined. The\nparent environment will be wherever the function happens to be called\nfrom. If a function f() is defined inside another function\ng() it will probably be used inside g() as well, so its\nparent environment and enclosing environment will probably be the same.\n

\n

Parent environments are important because things like model formulas\nneed to be evaluated in the environment the function was called from,\nsince that’s where all the variables will be available. This relies on\nthe parent environment being potentially different with each invocation.\n

\n

Enclosing environments are important because a function can use\nvariables in the enclosing environment to share information with other\nfunctions or with other invocations of itself (see the section on\nlexical scoping). This relies on the enclosing environment being the\nsame each time the function is invoked. (In C this would be done with\nstatic variables.)\n

\n

Scoping is hard. Looking at examples helps. It is particularly\ninstructive to look at examples that work differently in R and S and try\nto see why they differ. One way to describe the scoping differences\nbetween R and S is to say that in S the enclosing environment is\nalways the global workspace, but in R the enclosing environment\nis wherever the function was created.\n

\n
\n

7.13 How can I substitute into a plot label?

\n\n

Often, it is desired to use the value of an R object in a plot label,\ne.g., a title. This is easily accomplished using paste() if the\nlabel is a simple character string, but not always obvious in case the\nlabel is an expression (for refined mathematical annotation). In such a\ncase, either use parse() on your pasted character string or use\nsubstitute() on an expression. For example, if ahat is an\nestimator of your parameter a of interest, use\n

\n
\n
title(substitute(hat(a) == ahat, list(ahat = ahat)))\n
\n\n

(note that it is ‘==’ and not ‘=’). Sometimes bquote()\ngives a more compact form, e.g.,\n

\n
\n
title(bquote(hat(a) = .(ahat)))\n
\n\n

where subexpressions enclosed in ‘.()’ are replaced by their\nvalues.\n

\n

There are more examples in the mailing list archives.\n

\n

7.14 What are valid names?

\n\n

When creating data frames using data.frame() or\nread.table(), R by default ensures that the variable names are\nsyntactically valid. (The argument check.names to these\nfunctions controls whether variable names are checked and adjusted by\nmake.names() if needed.)\n

\n

To understand what names are “valid”, one needs to take into account\nthat the term “name” is used in several different (but related) ways\nin the language:\n

\n
    \n
  1. A syntactic name is a string the parser interprets as this type\nof expression. It consists of letters, numbers, and the dot and (for\nversions of R at least 1.9.0) underscore characters, and starts with\neither a letter or a dot not followed by a number. Reserved words are\nnot syntactic names.\n
  2. An object name is a string associated with an object that is\nassigned in an expression either by having the object name on the left\nof an assignment operation or as an argument to the assign()\nfunction. It is usually a syntactic name as well, but can be any\nnon-empty string if it is quoted (and it is always quoted in the call to\nassign()).\n\n
  3. An argument name is what appears to the left of the equals sign\nwhen supplying an argument in a function call (for example,\nf(trim=.5)). Argument names are also usually syntactic names,\nbut again can be anything if they are quoted.\n\n
  4. An element name is a string that identifies a piece of an object\n(a component of a list, for example.) When it is used on the right of\nthe ‘$’ operator, it must be a syntactic name, or quoted.\nOtherwise, element names can be any strings. (When an object is used as\na database, as in a call to eval() or attach(), the\nelement names become object names.)\n\n
  5. Finally, a file name is a string identifying a file in the\noperating system for reading, writing, etc. It really has nothing much\nto do with names in the language, but it is traditional to call these\nstrings file “names”.\n
\n\n

7.16 Why is the output not printed when I source() a file?

\n\n

Most R commands do not generate any output. The command\n

\n
\n
1+1\n
\n\n

computes the value 2 and returns it; the command\n

\n
\n
summary(glm(y~x+z, family=binomial))\n
\n\n

fits a logistic regression model, computes some summary information and\nreturns an object of class "summary.glm" (see How should I write summary methods?).\n

\n

If you type ‘1+1’ or ‘summary(glm(y~x+z, family=binomial))’ at\nthe command line the returned value is automatically printed (unless it\nis invisible()), but in other circumstances, such as in a\nsource()d file or inside a function it isn’t printed unless you\nspecifically print it.\n

\n

To print the value use\n

\n
\n
print(1+1)\n
\n\n

or\n

\n
\n
print(summary(glm(y~x+z, family=binomial)))\n
\n\n

instead, or use source(file, echo=TRUE).\n

\n

7.17 Why does outer() behave strangely with my function?

\n\n

As the help for outer() indicates, it does not work on arbitrary\nfunctions the way the apply() family does. It requires functions\nthat are vectorized to work elementwise on arrays. As you can see by\nlooking at the code, outer(x, y, FUN) creates two large vectors\ncontaining every possible combination of elements of x and\ny and then passes this to FUN all at once. Your function\nprobably cannot handle two large vectors as parameters.\n

\n

If you have a function that cannot handle two vectors but can handle two\nscalars, then you can still use outer() but you will need to wrap\nyour function up first, to simulate vectorized behavior. Suppose your\nfunction is\n

\n
\n
foo <- function(x, y, happy) {\n  stopifnot(length(x) == 1, length(y) == 1) # scalars only!\n  (x + y) * happy\n}\n
\n\n

If you define the general function\n

\n
\n
wrapper <- function(x, y, my.fun, ...) {\n  sapply(seq_along(x), FUN = function(i) my.fun(x[i], y[i], ...))\n}\n
\n\n

then you can use outer() by writing, e.g.,\n

\n
\n
outer(1:4, 1:2, FUN = wrapper, my.fun = foo, happy = 10)\n
\n\n

Scalar functions can also be vectorized using Vectorize().\n

\n
\n

7.18 Why does the output from anova() depend on the order of factors in the model?

\n\n

In a model such as ~A+B+A:B, R will report the difference in sums\nof squares between the models ~1, ~A, ~A+B and\n~A+B+A:B. If the model were ~B+A+A:B, R would report\ndifferences between ~1, ~B, ~A+B, and\n~A+B+A:B . In the first case the sum of squares for A is\ncomparing ~1 and ~A, in the second case it is comparing\n~B and ~B+A. In a non-orthogonal design (i.e., most\nunbalanced designs) these comparisons are (conceptually and numerically)\ndifferent.\n

\n

Some packages report instead the sums of squares based on comparing the\nfull model to the models with each factor removed one at a time (the\nfamous ‘Type III sums of squares’ from SAS, for example). These do not\ndepend on the order of factors in the model. The question of which set\nof sums of squares is the Right Thing provokes low-level holy wars on\nR-help from time to time.\n

\n

There is no need to be agitated about the particular sums of squares\nthat R reports. You can compute your favorite sums of squares quite\neasily. Any two models can be compared with anova(model1,\nmodel2), and drop1(model1) will show the sums of\nsquares resulting from dropping single terms.\n

\n

7.19 How do I produce PNG graphics in batch mode?

\n\n

Under a Unix-like, if your installation supports the\ntype="cairo" option to the png() device there should be no\nproblems, and the default settings should just work. This option is not\navailable for versions of R prior to 2.7.0, or without support for\ncairo. From R 2.7.0 png() by default uses the Quartz device\non macOS, and that too works in batch mode.\n

\n

Earlier versions of the png() device used the X11 driver, which\nis a problem in batch mode or for remote operation. If you have\nGhostscript you can use bitmap(), which produces a PostScript or\nPDF file then converts it to any bitmap format supported by Ghostscript.\nOn some installations this produces ugly output, on others it is\nperfectly satisfactory. Many systems now come with Xvfb from\nX.Org (possibly as an optional\ninstall), which is an X11 server that does not require a screen; and\nthere is the GDD package from CRAN, which produces PNG,\nJPEG and GIF bitmaps without X11.\n

7.22 Why do lattice/trellis graphics not work?

\n\n

The most likely reason is that you forgot to tell R to display the\ngraph. Lattice functions such as xyplot() create a graph object,\nbut do not display it (the same is true of ggplot2 graphics,\nand Trellis graphics in S-PLUS). The print() method for the\ngraph object produces the actual display. When you use these functions\ninteractively at the command line, the result is automatically printed,\nbut in source() or inside your own functions you will need an\nexplicit print() statement.\n

\n

7.31 Why doesn’t R think these numbers are equal?

\n\n

The only numbers that can be represented exactly in R’s numeric type are\nintegers and fractions whose denominator is a power of 2. All other\nnumbers are internally rounded to (typically) 53 binary digits accuracy.\nAs a result, two floating point numbers will not reliably be equal\nunless they have been computed by the same algorithm, and not always\neven then. For example\n

\n
\n
R> a <- sqrt(2)\nR> a * a == 2\n[1] FALSE\nR> a * a - 2\n[1] 4.440892e-16\nR> print(a * a, digits = 18)\n[1] 2.00000000000000044\n
\n\n

The function all.equal() compares two objects using a numeric\ntolerance of .Machine$double.eps ^ 0.5. If you want much greater\naccuracy than this you will need to consider error propagation\ncarefully.\n

\n

A discussion with many easily followed examples is in Appendix G\n“Computational Precision and Floating Point Arithmetic”, pages\n753–771 of Statistical Analysis and Data Display: An Intermediate\nCourse with Examples in R, Richard M. Heiberger and Burt Holland\n(Springer 2015, second edition). This appendix is a free download from\nhttp://link.springer.com/content/pdf/bbm%3A978-1-4939-2122-5%2F1.pdf.\n

\n

For more information, see e.g. David Goldberg (1991), “What Every\nComputer Scientist Should Know About Floating-Point Arithmetic”,\nACM Computing Surveys, 23/1, 5–48, also available via\nhttp://www.validlab.com/goldberg/paper.pdf.\n

\n

Here is another example, this time using addition:\n

\n
\n
R> .3 + .6 == .9\n[1] FALSE\nR> .3 + .6 - .9\n[1] -1.110223e-16\nR> print(matrix(c(.3,  .6,  .9, .3 + .6)), digits = 18)\n                     [,1]\n[1,] 0.299999999999999989\n[2,] 0.599999999999999978\n[3,] 0.900000000000000022\n[4,] 0.899999999999999911\n
\n\n\n

7.37 Why does backslash behave strangely inside strings?

\n\n

This question most often comes up in relation to file names (see\nHow do file names work in Windows?) but it also happens that\npeople complain that they cannot seem to put a single ‘\\’ character\ninto a text string unless it happens to be followed by certain other\ncharacters.\n

\n

To understand this, you have to distinguish between character strings\nand representations of character strings. Mostly, the\nrepresentation in R is just the string with a single or double quote at\neither end, but there are strings that cannot be represented that way,\ne.g., strings that themselves contain the quote character. So\n

\n
\n
> str <- "This \\"text\\" is quoted"\n> str\n[1] "This \\"text\\" is quoted"\n> cat(str, "\\n")\nThis "text" is quoted\n
\n\n

The escape sequences ‘\\"’ and ‘\\n’ represent a double\nquote and the newline character respectively. Printing text strings,\nusing print() or by typing the name at the prompt will use the\nescape sequences too, but the cat() function will display the\nstring as-is. Notice that ‘"\\n"’ is a one-character string, not\ntwo; the backslash is not actually in the string, it is just generated\nin the printed representation.\n

\n
\n
> nchar("\\n")\n[1] 1\n> substring("\\n", 1, 1)\n[1] "\\n"\n
\n\n

So how do you put a backslash in a string? For this, you have to\nescape the escape character. I.e., you have to double the backslash.\nas in\n

\n
\n
> cat("\\\\n", "\\n")\n\\n\n
\n\n

Some functions, particularly those involving regular expression\nmatching, themselves use metacharacters, which may need to be escaped by\nthe backslash mechanism. In those cases you may need a quadruple\nbackslash to represent a single literal one.\n

\n

In versions of R up to 2.4.1 an unknown escape sequence like ‘\\p’\nwas quietly interpreted as just ‘p’. Current versions of R emit a\nwarning.\n

\n

7.41 Why does summary() report strange results for the R^2 estimate when I fit a linear model with no intercept?

\n\n

As described in ?summary.lm, when the intercept is zero (e.g.,\nfrom y ~ x - 1 or y ~ x + 0), summary.lm() uses the\nformula\n R^2 = 1 - Sum(R[i]^2) / Sum((y[i])^2)\nwhich is different from the usual\n R^2 = 1 - Sum(R[i]^2) / Sum((y[i] - mean(y))^2).\nThere are several reasons for this:\n

    \n
  • Otherwise the R^2 could be negative (because the model with zero\nintercept can fit worse than the constant-mean model it is\nimplicitly compared to).\n
  • If you set the slope to zero in the model with a line through the\norigin you get fitted values y*=0\n
  • The model with constant, non-zero mean is not nested in the model\nwith a line through the origin.\n
\n\n

All these come down to saying that if you know a priori that\nE[Y]=0 when x=0 then the ‘null’ model that you should\ncompare to the fitted line, the model where x doesn’t explain any\nof the variance, is the model where E[Y]=0 everywhere. (If you\ndon’t know a priori that E[Y]=0 when x=0, then you\nprobably shouldn’t be fitting a line through the origin.)\n

\n

7.42 Why is R apparently not releasing memory?

\n\n

This question is often asked in different flavors along the lines of\n“I have removed objects in R and run gc() and yet\nps/top still shows the R process using a lot of\nmemory”, often on Linux machines.\n

\n

This is an artifact of the way the operating system (OS) allocates\nmemory. In general it is common that the OS is not capable of\nreleasing all unused memory. In extreme cases it is possible that even\nif R frees almost all its memory, the OS can not release any of it due\nto its design and thus tools such as ps or top will\nreport substantial amount of resident RAM used by the R process even\nthough R has released all that memory. In general such tools do\nnot report the actual memory usage of the process but rather\nwhat the OS is reserving for that process.\n

\n

The short answer is that this is a limitation of the memory allocator\nin the operating system and there is nothing R can do about it. That\nspace is simply kept by the OS in the hope that R will ask for it\nlater. The following paragraph gives more in-depth answer with\ntechnical details on how this happens.\n

\n

Most systems use two separate ways to allocate memory. For allocation\nof large chunks they will use mmap to map memory into the\nprocess address space. Such chunks can be released immediately when\nthey are completely free, because they can reside anywhere in the\nvirtual memory. However, this is a relatively expensive operation and\nmany OSes have a limit on the number of such allocated chunks, so this\nis only used for allocating large memory regions. For smaller\nallocations the system can expand the data segment of the process\n(historically using the brk system call), but this whole area\nis always contiguous. The OS can only move the end of this space, it\ncannot create any “holes”. Since this operation is fairly cheap, it\nis used for allocations of small pieces of memory. However, the\nside-effect is that even if there is just one byte that is in use\nat the end of the data segment, the OS cannot release any memory\nat all, because it cannot change the address of that byte. This is\nactually more common than it may seem, because allocating a lot of\nintermediate objects, then allocating a result object and removing all\nintermediate objects is a very common practice. Since the result is\nallocated at the end it will prevent the OS from releasing any memory\nused by the intermediate objects. In practice, this is not necessarily\na problem, because modern operating systems can page out unused\nportions of the virtual memory so it does not necessarily reduce the\namount of real memory available for other applications. Typically,\nsmall objects such as strings or pairlists will be affected by this\nbehavior, whereas large objects such as long vectors will be allocated\nusing mmap and thus not affected. On Linux (and possibly other\nUnix-like systems) it is possible to use the mallinfo system call\n(also see the mallinfo package) to\nquery the allocator about the layout of the allocations, including the\nactually used memory as well as unused memory that cannot be released.\n

\n
\n

7.43 How can I enable secure https downloads in R?

\n\n\n

When R transfers files over HTTP (e.g., using the\ninstall.packages() or download.file() function), a\ndownload method is chosen based on the download.file.method\noption. There are several methods available and the default behavior if\nno option is explicitly specified is to use R’s internal HTTP\nimplementation. In most circumstances this internal method will not\nsupport HTTPS URLs so you will need to override the default:\nthis is done automatically for such URLs as from R 3.2.2.\n

\n

R versions 3.2.0 and greater include two download methods\n("libcurl" and "wininet") that both support\nHTTPS connections: we recommend that you use these methods.\nThe requisite code to add to .Rprofile or Rprofile.site is:\n

\n
\n
options(download.file.method = "wininet", url.method = "wininet")\n(Windows)\noptions(download.file.method = "libcurl", url.method = "libcurl")\n(Linux and macOS)\n
\n\n

(Method "wininet" is the default on Windows as from R 3.2.2.)\n

\n

Note that the "libcurl" method may or may not have been compiled\nin. In the case that it was not, i.e. capabilities("libcurl") ==\nFALSE, we recommend method "wget" on Linux and "curl" on\nmacOS. It is possible that system versions of "libcurl",\nwget or curl may have been compiled without\nHTTPS support, but this is unlikely. As from R 3.3.0\n"libcurl" with HTTPS support is required except on\nWindows.\n

Suppose you have C code file for dynloading into R, but you want to use\nR CMD SHLIB with compilation flags other than the default ones\n(which were determined when R was built).\n

\n

Starting with R 2.1.0, users can provide personal Makevars configuration\nfiles in $HOME/.R to override the default flags.\nSee section “Add-on packages” in R Installation and\nAdministration.\n

\n

For earlier versions of R, you could change the file\nR_HOME/etc/Makeconf to reflect your preferences, or (at\nleast for systems using GNU Make) override them by the\nenvironment variable MAKEFLAGS.\nSee section “Creating shared objects” in Writing R Extensions.\n

9.1 What is a bug?

\n\n

If R executes an illegal instruction, or dies with an operating system\nerror message that indicates a problem in the program (as opposed to\nsomething like “disk full”), then it is certainly a bug. If you call\n.C(), .Fortran(), .External() or .Call() (or\n.Internal()) yourself (or in a function you wrote), you can\nalways crash R by using wrong argument types (modes). This is not a\nbug.\n

\n

Taking forever to complete a command can be a bug, but you must make\ncertain that it was really R’s fault. Some commands simply take a long\ntime. If the input was such that you know it should have been\nprocessed quickly, report a bug. If you don’t know whether the command\nshould take a long time, find out by looking in the manual or by asking\nfor assistance.\n

\n

If a command you are familiar with causes an R error message in a case\nwhere its usual definition ought to be reasonable, it is probably a bug.\nIf a command does the wrong thing, that is a bug. But be sure you know\nfor certain what it ought to have done. If you aren’t familiar with the\ncommand, or don’t know for certain how the command is supposed to work,\nthen it might actually be working right. For example, people sometimes\nthink there is a bug in R’s mathematics because they don’t understand\nhow finite-precision arithmetic works. Rather than jumping to\nconclusions, show the problem to someone who knows for certain.\nUnexpected results of comparison of decimal numbers, for example\n0.28 * 100 != 28 or 0.1 + 0.2 != 0.3, are not a bug.\nSee Why doesn\'t R think these numbers are equal?, for more details.\n

\n

Finally, a command’s intended definition may not be best for statistical\nanalysis. This is a very important sort of problem, but it is also a\nmatter of judgment. Also, it is easy to come to such a conclusion out\nof ignorance of some of the existing features. It is probably best not\nto complain about such a problem until you have checked the\ndocumentation in the usual ways, feel confident that you understand it,\nand know for certain that what you want is not available. If you are\nnot sure what the command is supposed to do after a careful reading of\nthe manual this indicates a bug in the manual. The manual’s job is to\nmake everything clear. It is just as important to report documentation\nbugs as program bugs. However, we know that the introductory\ndocumentation is seriously inadequate, so you don’t need to report this.\n

\n

If the online argument list of a function disagrees with the manual, one\nof them must be wrong, so report the bug.\n

9.2 How to report a bug

\n\n

When you decide that there is a bug, it is important to report it and to\nreport it in a way which is useful. What is most useful is an exact\ndescription of what commands you type, starting with the shell command\nto run R, until the problem happens. Always include the version of R,\nmachine, and operating system that you are using; type version in\nR to print this.\n

\n

The most important principle in reporting a bug is to report\nfacts, not hypotheses or categorizations. It is always easier to\nreport the facts, but people seem to prefer to strain to posit\nexplanations and report them instead. If the explanations are based on\nguesses about how R is implemented, they will be useless; others will\nhave to try to figure out what the facts must have been to lead to such\nspeculations. Sometimes this is impossible. But in any case, it is\nunnecessary work for the ones trying to fix the problem.\n

\n

For example, suppose that on a data set which you know to be quite large\nthe command\n

\n
\n
R> data.frame(x, y, z, monday, tuesday)\n
\n\n

never returns. Do not report that data.frame() fails for large\ndata sets. Perhaps it fails when a variable name is a day of the week.\nIf this is so then when others got your report they would try out the\ndata.frame() command on a large data set, probably with no day of\nthe week variable name, and not see any problem. There is no way in the\nworld that others could guess that they should try a day of the week\nvariable name.\n

\n

Or perhaps the command fails because the last command you used was a\nmethod for "["() that had a bug causing R’s internal data\nstructures to be corrupted and making the data.frame() command\nfail from then on. This is why others need to know what other commands\nyou have typed (or read from your startup file).\n

\n

It is very useful to try and find simple examples that produce\napparently the same bug, and somewhat useful to find simple examples\nthat might be expected to produce the bug but actually do not. If you\nwant to debug the problem and find exactly what caused it, that is\nwonderful. You should still report the facts as well as any\nexplanations or solutions. Please include an example that reproduces\n(e.g., https://en.wikipedia.org/wiki/Reproducibility) the problem,\npreferably the simplest one you have found.\n

\n

Invoking R with the --vanilla option may help in isolating a\nbug. This ensures that the site profile and saved data files are not\nread.\n

\n

Before you actually submit a bug report, you should check whether the\nbug has already been reported and/or fixed. First, try the “Show open\nbugs new-to-old” or the search facility on\nhttps://bugs.R-project.org/. Second, consult\nhttps://svn.R-project.org/R/trunk/doc/NEWS.Rd, which\nrecords changes that will appear in the next release of R,\nincluding bug fixes that do not appear on the Bug Tracker. \nThird, if possible try the current r-patched or r-devel version of R.\nIf a bug has already been reported or fixed, please do not submit\nfurther bug reports on it. Finally, check carefully whether the bug is\nwith R, or a contributed package. Bug reports on contributed packages\nshould be sent first to the package maintainer, and only submitted to\nthe R-bugs repository by package maintainers, mentioning the package in\nthe subject line.\n

\n

A bug report can be generated using the function bug.report().\nFor reports on R this will open the Web page at\nhttps://bugs.R-project.org/: for a contributed package it will open\nthe package’s bug tracker Web page or help you compose an email to the\nmaintainer.\n

\n

There is a section of the bug repository for suggestions for\nenhancements for R labelled ‘wishlist’. Suggestions can be\nsubmitted in the same ways as bugs, but please ensure that the subject\nline makes clear that this is for the wishlist and not a bug report, for\nexample by starting with ‘Wishlist:’.\n

\n

Comments on and suggestions for the Windows port of R should be sent to\nR-windows@R-project.org.\n

\n

Corrections to and comments on message translations should be sent to the\nlast translator (listed at the top of the appropriate ‘.po’ file)\nor to the translation team as listed at\nhttps://developer.R-project.org/TranslationTeams.html.\n
